## 1. 서비스 목적 및 타겟 유저 정의

### 1.1 서비스 목적

**핵심 목적:**
- 개인 브랜드(개발자/창업자)와 제품(CodeForge)을 **통합적으로 소개**하여 신뢰도 구축
- 잠재 고객의 의사결정을 가속화하고, 투자자/파트너와의 접점 창출
- SEO를 통한 **자연 유입**과 **마케팅 비용 절감**

**Why this matters:**
1인 개발창업자는 개인의 능력과 제품의 신뢰성이 강하게 연결되어 있음. 따라서 "누가 만들었는가"가 "무엇을 만들었는가" 만큼 중요한 의사결정 요소가 됨.

### 1.2 타겟 유저 정의

| 세그먼트 | 비중 | 주요 니즈 | 행동 패턴 |
|---------|------|---------|---------|
| **B2B 잠재 고객** | 30% | 제품 기능/가격/보증 이해 | 제품 페이지 → 가격/문서 → 문의 |
| **B2C 잠재 고객** | 20% | 사용 사례/후기 확인 | Hero → 사례 → CTA → 가입 |
| **초기 투자자** | 15% | 창업자 역량/비전/성과 증명 | About → 경력/프로젝트 → Contact |
| **기술 파트너** | 10% | 기술 스택/아키텍처/협업 가능성 | Skills/Stack → 프로젝트 → Contact |
| **채용 공고 지원자** | 15% | 회사 문화/기술/성장성 | About → 블로그(Tech Insights) → Contact |
| **잠재 협업 기관** | 10% | 브랜드/신뢰성/포트폴리오 | 전체 사이트 (신뢰도 검증) |

**핵심 인사이트:**
- 각 세그먼트별로 **진입 경로와 관심사가 상이함** → 섹션 구성 시 모든 진입점에서 가치를 제공해야 함
- 투자자와 고객이 같은 사이트를 보지만, **강조 포인트가 다름** → 콘텐츠 우선순위화 필요

## 2. 핵심 메시지 & 브랜딩 전략

### 2.1 브랜드 포지셔닝

```
"AI 시대의 개발 생산성을 혁신하는 개발자 창업자"
(A developer-founder innovating development productivity in the AI era)
```

**포지셔닝 이유:**
- 기술 신뢰성 강조 (AI 기술 리더십)
- 실행력 강조 (창업 = 실제 행동 증명)
- 시대 감각 강조 (AI 트렌드 리더)

### 2.2 핵심 메시지 전략

| 페르소나 | 핵심 메시지 | 서브메시지 |
|---------|-----------|----------|
| **B2B/B2C 고객** | "코드 작성 시간을 70% 단축하세요" | 검증된 AI 기술 + 실제 사용자 사례 |
| **투자자** | "개발 생산성 시장에서 첫 번째 무브 메이커" | 창업자 역량 증명 + 성장 데이터 |
| **기술 파트너** | "차세대 개발 도구 생태계의 핵심 인프라" | 기술 뛰어남 + 개방적 협력 의지 |
| **채용 관심사** | "기술 문화 있는 스타트업" | 성장 기회 + 기술 리더와의 협업 |

### 2.3 브랜드 성격 (Tone & Manner)

- **Professional but Approachable**: 기술 전문성 + 개인의 인간성
- **Action-Oriented**: "계획"보다 "실행"과 "결과" 강조
- **Transparent**: 실패 경험, 기술 도전 과정도 솔직하게 공유
- **Data-Driven**: 추측이 아닌 수치/증거 기반 주장

## 3. 페이지 구성 (정보구조 - IA)

### 3.1 사이트맵 (메인 네비게이션)

```
홈 (Home)
├─ Hero Section
├─ Featured Product/Project
└─ CTA

소개 (About)
├─ 개인 스토리 (Why I built this)
├─ 경력 하이라이트 (Timeline)
└─ 현재 포커스

제품 (Products)
├─ CodeForge (주력 제품)
│  ├─ 개요
│  ├─ 기능
│  ├─ 사용 사례
│  ├─ 가격
│  └─ CTA (Free Trial)
└─ (향후) 2번째 제품

프로젝트 (Projects)
├─ 오픈소스 기여
├─ 개인 사이드 프로젝트
└─ 학습 자료/튜토리얼

기술 & 스택 (Skills & Stack)
├─ 핵심 기술 영역
├─ 사용 중인 도구/라이브러리
└─ 기술 블로그 링크

블로그 (Blog) [v1.1에서 추가]
├─ 기술 인사이트
├─ 창업 일지
└─ 산업 분석

문의 (Contact)
├─ 이메일 폼
├─ 소셜 링크
└─ 일정 예약 (Calendly)
```

### 3.2 깊이(Depth) 전략

| 페이지 | Depth | 이유 |
|--------|-------|------|
| Home | 1 | 모든 주요 세그먼트로의 진입점 |
| About | 2 | 깊이 있는 스토리텔링, 타겟별 메시지 분화 |
| Products (CodeForge) | 3 | 제품 상세 → 기능 → 사용 사례 → 가격 |
| Projects | 2 | 개별 프로젝트 페이지는 외부 링크 |
| Contact | 1 | 접근성 최우선 |

**모바일 고려사항:**
- 모든 페이지 Depth 제한: 3 이하 (스크롤 과다 방지)
- 탭 네비게이션: Home, About, Products, Projects, Contact (5개 탭)
- 하단 고정 CTA 바 (제품 페이지에서)

**예외 케이스:**
- 이미 CodeForge 사용자: 제품 페이지로 바로 이동하도록 옵션 제공
- 모바일 사용자: 이미지 최소화, 텍스트 중심으로 재배치

- **UX 전략**:
  - 제품 페이지에서 **기능 비교표** 강조
  - 3분 데모 비디오 제공
  - 가격 명확히 표시
  - 평가/후기 섹션

- **행동 패턴**: 링크 → About 섹션 정독 → 경력 확인 → Contact
- **구매 결정 요소**: 창업자 역량 + 시장 규모 + 성장 데이터
- **UX 전략**:
  - About에서 **경력 Timeline** 강조
  - 데이터/지표 중심 설명 (추측 X)
  - 연락처 명확히 제공

#### Persona C: "기술 커뮤니티 개발자"

### 7.2 UX 설계 원칙

#### 원칙 1: "한눈에 이해하기"
- 각 섹션 상단에서 핵심 메시지를 **3초 안에 파악** 가능하게 설계
- 긴 텍스트보다 **시각화 (표, 아이콘, 그래프)** 활용

#### 원칙 2: "명확한 CTA 계층화"
- Primary CTA (제품 가입): 5-6군데
- Secondary CTA (정보 제공): 각 섹션별 1-2개
- Tertiary CTA (외부 링크): GitHub, LinkedIn 등

#### 원칙 3: "신뢰도 신호 배치"
- 각 섹션에 신뢰도 신호 배치:
  - Hero: 사용자 수, 시간 단축율
  - About: 경력, 경험
  - Products: 가격 투명성, 평가
  - Projects: GitHub 별, 사용자 수
  - Contact: 빠른 응답 보장

#### 원칙 4: "퍼소나별 진입 경로 최적화"
- 고객 → Products 직진로
- 투자자 → About 직진로
- 개발자 → Projects 직진로
- 각 진입로에서 최소 2-3클릭 안에 핵심 정보 도달

#### 원칙 5: "모바일 우선"
- 모든 섹션을 스크롤로 이동 (가로스크롤 X)
- 터치 타겟 최소 44x44px 준수
- 이미지 로딩 최적화 (WebP + 프로그레시브)

## 8. 기술 스택 제안

### 8.1 기존 스택 활용 (권장)

```
현재 스택:
├─ Framework: Next.js 13+ (App Router)
├─ Styling: Tailwind CSS + CSS Module
├─ i18n: next-intl (EN / KO)
├─ Icons: Lucide React
├─ Hosting: Vercel (기존 CodeForge 페이지)
└─ Analytics: Google Analytics 4

├─ Form: react-hook-form (Contact 폼)
```

**선택 사유:**
- 기존 개발 환경과 일치 → 학습곡선 최소화
- Next.js의 SSG 지원 → SEO 최적화
- Tailwind의 빠른 개발 속도 → MVP 2.5주 달성 가능
- Vercel 호스팅 → 국내/해외 최적 성능

| **이메일** | Resend | Next.js 최적화, React 컴포넌트 지원 |
| **분석** | Vercel Analytics | 기본 제공, 별도 설정 불필요 |
| **일정 관리** | Calendly (임베드) | 외부 서비스, 별도 개발 불필요 |
| **이미지 CDN** | Cloudinary (선택) | 이미지 최적화 (제품 스크린샷) |

## 10. 마이그레이션 및 런칭 계획

### 10.1 기존 CodeForge 페이지와의 관계

**옵션 A: 통합 (권장)**
```
홈페이지 구조 (통합):
├─ homepage.com/
│  ├─ / (개인 브랜딩 홈)
│  ├─ /about (개인 소개)
│  ├─ /products/codeforge (CodeForge 상세)
│  ├─ /projects
│  └─ /contact
│
└─ codeforge.com → homepage.com/products/codeforge 리다이렉트
```

**장점:**
- 통일된 도메인
- SEO 권력 집중
- 관리 편의성

**단점:**
- 기존 CodeForge 트래픽 영향 가능

**옵션 B: 병행 (보수적)**
```
홈페이지: homepage.com/
CodeForge: codeforge.com/ (기존 유지)

홈페이지에서 CodeForge 링크로 연결
```

**추천:** 옵션 A (개인 브랜드 강화)

## 13. 용어 정의 (Glossary)

| 용어 | 정의 |
|------|------|
| **Hero Section** | 웹 페이지 최상단의 첫 화면 영역 (사용자가 도착했을 때 보이는 부분) |
| **CTR (Click-Through Rate)** | 노출된 요소 중 실제 클릭된 비율 |
| **Conversion Rate** | 방문자 중 특정 행동(가입, 구매 등)을 한 비율 |
| **CPA (Cost Per Acquisition)** | 고객 1명을 획득하는 데 드는 마케팅 비용 |
| **Core Web Vitals** | Google이 중요하게 평가하는 성능 지표 (로딩 시간, 상호작용, 안정성) |
| **Schema.org** | 검색엔진이 이해할 수 있는 구조화된 데이터 형식 |
| **MVP (Minimum Viable Product)** | 최소한의 기능으로 만든 제품 |
| **UX (User Experience)** | 사용자가 제품을 사용하며 느끼는 전체 경험 |
| **A/B Testing** | 두 가지 버전을 비교하여 어느 것이 더 효과적인지 검증하는 방법 |
